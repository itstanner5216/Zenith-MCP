# Detection/Scoping Mechanism Inventory

Every distinct "what project am I in?" mechanism that has existed in Zenith-MCP,
deduped across eras. See section 3 for the how-they-fit-together chain and the
current gaps.

## 1. Deduped Mechanism List

| # | Mechanism | What it does | Where it lives (best incarnation) | Status | Known bugs/caveats |
|---|---|---|---|---|---|
| 1 | **Explicit binding (`initProject`)** | User/tool manually says "this directory is my project." Binds immediately and is "sticky" — nothing else overrides it for the rest of the session. | `project-context.ts` `initProject()` (current, in-memory only). Earlier versions (ladder, JS) persisted the binding to a SQLite `project_roots` table so it survived reconnects. | LIVE | Current version dropped SQLite persistence — a manual `initProject` no longer survives a server restart, only a config-file entry does. |
| 2 | **Config-file project registry** | Reads a `### Projects` section from `~/.zenith-mcp/config` (project id, name, root, tags, include/exclude, etc.) and matches incoming file paths against it by id, name, path-segment, or path-prefix. This is the current sole source of "known projects." | `project-registry.ts` (`ProjectRegistry`) + `project-context.ts` `reloadRegistry()` / `_tryLazyReload()` | LIVE | Registry's `findProject` prefix-match `reduce` compares raw (non-normalized) `manifest.project_root` string lengths instead of the normalized key, so the "longest/most specific root wins" tie-break can pick the wrong entry when path casing/trailing-slash differs (audit P2). |
| 3 | **SQLite-backed project registry** | Same matching logic as #2, but the list of known projects came from a `project_roots` table in the global SQLite DB (populated only by `initProject`), auto-loaded into memory via `_syncRegistry()` on construction/refresh. | Ladder era: `project-context.LAST-WIRED-LADDER.ts` `_syncRegistry()`; JS original: `project-context.js` `_resolveFromRegistry()` (simpler `cwd.startsWith(root)` check, no `ProjectRegistry` class yet). | HISTORICAL | `_syncRegistry()` swallows all errors silently (audit P3) — a corrupted/missing DB makes every registered project invisible with no warning. Superseded by #2 (config file is now authoritative; SQLite registry entries are explicitly NOT merged per current `project-context.ts` comment). |
| 4 | **Git repo root detection (`findRepoRoot`)** | Walks up from a file/dir looking for a `.git` file or folder using plain filesystem stat calls (no shelling out required), then optionally shells out to `git rev-parse --show-toplevel` for accuracy on worktrees/submodules, falling back to the `.git` parent if the git CLI is missing or fails. | `symbol-index.ts` lines 40-87 (current, unchanged from the fix-plan's recommended version) | LIVE (function exists and is called from tools), but **not used for project-context resolution** — `ProjectContext` no longer calls it at all; it's only called directly by `edit_file.ts` and `stash_restore.ts`. | Fix-plan Issue 6 predates this filesystem-first version: the original had git-CLI-only detection with no PATH fallback, so it silently failed on GUI-launched clients with minimal PATH. That's fixed in the current code. |
| 5 | **Marker-file detection (`_resolveFromMarkers`)** | Walks up from a file looking for language/build markers (`package.json`, `Cargo.toml`, `go.mod`, `pyproject.toml`, `pom.xml`, etc. — 15 markers including `.git`), skipping over `node_modules`/`dist`/`build`/etc. Inside a git repo it returns the *deepest* marker (handles monorepo packages); outside a git repo it returns the *nearest* marker walking up. | `recovered-project-scoping/project-scope.ts` `_resolveFromMarkers()` (peak TS ladder version, shares the git root as a "ceiling" to avoid a second `findRepoRoot` call) | ORPHANED-RECOVERED | Audit P2: when the git-repo ceiling is reached with zero markers found between the file and the ceiling, the function can return `null` even though the ceiling itself may hold a marker, because the ceiling directory's parent boundary check has an edge case. Also never scans the filesystem root itself. Fix-plan Issue 7 notes the earlier version called `findRepoRoot` a second time just to compute the ceiling, doubling git-CLI latency/timeout risk — already fixed in the recovered version by threading the raw git root through. |
| 6 | **MCP roots / allowed-directories fallback (`_resolveFromAllowedDirectories`)** | When file-based git/marker detection fails, tries the directories the MCP client (or CLI args) said are "allowed": checks each for a git repo, and if there's exactly one allowed directory, uses it outright. | `recovered-project-scoping/project-scope.ts` `_resolveFromAllowedDirectories()` | ORPHANED-RECOVERED | Audit P2: uses its own ad-hoc clamping logic instead of the shared `clampToAllowed()` helper used by Steps 1-2, so the same path can resolve differently in Step 3 vs Steps 1-2 depending on which step "wins." |
| 7 | **Path-clamping to allowed directories (`clampToAllowed` / `getMostSpecificAllowedRoot`)** | Safety net so that a detected git/marker root never resolves to somewhere *outside* the client-approved sandbox directories — if the detected root contains the allowed dir, it clamps down to the allowed dir; if it's inside, keeps it as-is. | `recovered-project-scoping/project-scope.ts` lines 100-127 | ORPHANED-RECOVERED | Audit P2: with a single narrow allowed directory (e.g. `/repo/packages/app`) and a wider git root (`/repo`), clamping silently narrows the project to the subpackage, which may surprise users who expected the whole repo. Secure but not always what's wanted. |
| 8 | **Resolution result caching** | LRU cache (512 entries) keyed on absolute path + sorted allowed dirs + sorted registry roots, so repeated resolutions for the same file/config state skip re-walking the filesystem. | `recovered-project-scoping/project-scope.ts` `_cache` / `getCached` / `setCached` / `clearProjectScopeCache()` | ORPHANED-RECOVERED | Audit P3: cache key includes full registry fingerprint, so every `initProject()` call invalidates effectively the whole cache (stale entries just sit there until LRU-evicted, not actually cleared) — `initProject()` never calls `clearProjectScopeCache()`. |
| 9 | **Process-tree CWD walk** | Reads `/proc/<pid>/cwd` for parent, grandparent, etc. (up to 5 ancestors) to find the working directory of the shell/IDE that launched the MCP server, since the server's own `process.cwd()` is usually wrong (e.g. `/` or `$HOME`). macOS gets a weaker `lsof`-based variant; Windows has no equivalent and just falls back to the server's own cwd. | `recovered-project-scoping/process-tree.ts` (full implementation) + wired into `project-context.LAST-WIRED-LADDER.ts` `_resolve()` Priority 2 | ORPHANED-RECOVERED | This was fix-plan Issue 14, flagged CRITICAL ("no process-tree CWD walk" = can't find project when roots/CLI unavailable). The recovered file is a complete, working implementation but nothing in current `src/` imports it. |
| 10 | **cwd-based resolution (`process.cwd()` fallback)** | Falls back to treating the server process's own current working directory as a resolution candidate, run through the same git/marker/registry logic. | JS original: `project-context.js` `_resolve()` Step 4; ladder version keeps it as the tail candidate after process-tree walk. | HISTORICAL | Fix-plan Issue 2 (CRITICAL): the server's `process.cwd()` is essentially never the user's project (it's wherever the client app happened to launch the process from), so this step almost always returns null in practice. This is exactly why the process-tree walk (#9) was invented — cwd alone isn't reliable. |
| 11 | **MCP-roots-based auto-switch (registry-first)** | On every file access, checks whether the path is still inside the currently bound root; if not, looks it up in the config registry and switches the "active project" transparently, notifying the client once per unrecognized root. | `project-context.ts` `_handlePathAccess()` (current) | LIVE | None noted beyond general registry caveats (#2). This replaces the ladder's git/marker auto-promote (#12) entirely — it is registry-only, deliberately no git/marker fallback. |
| 12 | **Auto-promote first-touched root as session binding** | The first file path resolved successfully (via whichever detection ran) becomes the session's "sticky" bound root for subsequent calls that don't pass a file path — as long as nothing was explicitly bound. | JS original + ladder version, `getRoot()` | HISTORICAL | Audit P2: condition `!this._resolved || !this._boundRoot` re-fires auto-promote even after a prior resolution deliberately decided "global" (null), silently overwriting that decision on the next file-scoped call. Also: in multi-project workspaces, whichever file happens to be touched *first* locks the whole session to that project, which can be surprising. Not present in current code — current registry-only design binds per-access instead (#11), sidestepping this specific bug. |
| 13 | **Global fallback (`~/.zenith-mcp/global-stash.db`)** | When nothing else resolves a project, use one shared global database instead of failing. This is the universal safety net across every era of the code. | All versions: `project-context.ts` (current), ladder, and JS original all implement `getGlobalDb()` / `getStashDb()` global path. | LIVE | Fix-plan Issues 1, 3, 4 (all CRITICAL, ladder era only) describe how bugs elsewhere (crash-on-init, empty-allowed-dirs throwing, an overly aggressive guard) prevented the code from ever *reaching* this fallback even though it existed. Those bugs are not present in the current registry-only design. |
| 14 | **`onRootsChanged` refresh hook** | Notifies the `ProjectContext` singleton (per session) that the MCP client's declared roots changed, so it should drop cached state and re-resolve — except explicit (`initProject`) bindings, which stay sticky. | `project-context.ts` `onRootsChanged()` / `refresh()` (current — `ctx` is a required param) | LIVE | Historical JS version had `ctx` optional and silently no-op'd if omitted (fix-plan Issue 13, CRITICAL — "silent no-op sabotage"); already fixed by making `ctx` required in both the ladder and current versions. |
| 15 | **Session-root hint registration (`registerSessionRoot`)** | Lets the server register an MCP root as a lightweight "hint" for the session without making it a sticky explicit binding — only takes effect if it happens to match a project already in the registry. | `project-context.ts` `registerSessionRoot()` (current) | LIVE | No caveats found; this is new to the current registry-only design and has no equivalent in the recovered ladder/JS versions. |
| 16 | **Nearest-projectish-root walk for notification dedup** | Walks up from an unrecognized path looking for `.git`/`package.json`/etc. **purely to pick a stable, human-readable name for a "not a registered project" notification** — explicitly NOT used to actually decide the project root. | `project-context.ts` `_findNearestProjectishRoot()` (current) | LIVE | This looks like marker detection (#5) but is intentionally cosmetic/notification-only in the current design — worth noting so nobody mistakes it for a real detection path when reintegrating. |
| 17 | **Process-tree walk as a per-file resolution step (not just session fallback)** | Same `/proc` walk as #9, but proposed as "Step 0" *inside* `resolveProjectRoot()` itself so a single file lookup (not just whole-session resolution) benefits when the file's own path is outside all allowed dirs. | Described only in `docs/project-scoping/project-scoping-fix-plan.md` Issue 14 ("Also add process-tree walk to resolveProjectRoot as Step 0") | PLANNED-ONLY | Never implemented anywhere, including the recovered ladder file — the ladder only wired the process-tree walk into `ProjectContext._resolve()` (session level), not into `resolveProjectRoot()` (per-file level). |
| 18 | **Merge (not replace) MCP roots with CLI-provided allowed dirs** | Proposed fix so that when an MCP client sends its roots, they're unioned with the server's CLI-provided `_allowedDirectories` instead of wholesale replacing them (which currently makes CLI dirs vanish the moment a client connects). | `docs/project-scoping/project-scoping-fix-plan.md` Issue 5 | PLANNED-ONLY | Not detection per se, but directly gates which directories detection can ever see — worth reintegrating alongside any detection mechanism that depends on allowed dirs. |
| 19 | **Seed registry from MCP root `name` field** | Proposed: when a client sends roots with human-readable `name`s, automatically call `initProject(path, name)` to seed the registry, instead of leaving `name` as unused dead data. | `docs/project-scoping/project-scoping-fix-plan.md` Issue 12 | PLANNED-ONLY | Never implemented. Would give the config-registry / explicit-binding mechanisms (#1, #2) an automatic population source instead of requiring manual config edits. |

Not counted as separate mechanisms (implementation details, not detection strategies):
LRU cache internals, path normalization (`normalizePath`), path-containment checks (`isWithinProject` / `isPathWithinAllowedDirectories` — audited as inconsistent in P2 but not a detection mechanism), and MCP roots URI parsing (`parseRootUri`) — these all support detection but don't themselves decide "what project am I in."

## 2. How They'd Fit Together — Resolution Order

The peak-TS ladder era (`project-context.LAST-WIRED-LADDER.ts` + `project-scope.ts` + `process-tree.ts`) designed one explicit chain, run top to bottom until something matches:

```
0. Process-tree cwd walk         (#9)   -- planned as session-level AND per-file Step 0 (#17, only session-level ever wired)
1. Git repo detection            (#4)   -- from the file path itself
2. Marker-based detection        (#5)   -- deepest marker within git root, or nearest without git
3. Allowed-directories fallback  (#6)   -- MCP roots / CLI dirs, clamped (#7)
4. Registry matching             (#2/#3)-- SQLite-backed registry in the ladder era
5. Global fallback               (#13)  -- ~/.zenith-mcp/global-stash.db
```
(Explicit binding via `initProject`, #1, always short-circuits everything above when set — it's sticky.)

**Current live code status against that chain:**

| Link | Live today? |
|---|---|
| 0. Process-tree walk | **Missing** — recovered file exists (#9) but nothing imports it. |
| 1. Git detection | **Missing from ProjectContext** — `findRepoRoot` (#4) still exists and is called directly by `edit_file.ts`/`stash_restore.ts`, but `ProjectContext` itself never calls it. |
| 2. Marker detection | **Missing** — orphaned in `project-scope.ts` (#5), not called from anywhere in current `src/`. |
| 3. Allowed-dirs fallback | **Missing as a resolution step** — allowed dirs are only used today to pick which registry entry matches (`_resolveNoFile()`), not as a detection source in their own right. |
| 4. Registry matching | **Live, but redesigned** — replaced by the config-file registry (#2), which is now the *primary and first* automatic mechanism instead of a last-resort. |
| 5. Global fallback | **Live and reachable** — none of the ladder-era crash bugs (fix-plan Issues 1/3/4) exist in the current code. |
| Explicit `initProject` binding | **Live** (#1), though no longer SQLite-persisted. |

In short: the current design deliberately deleted the automatic (git/marker/process-tree) part of the chain and kept only explicit binding → config registry → global fallback, on the theory (per the code comment) that automatic detection was a source of bugs and inconsistency. Everything upstream of the registry in the old ladder is now orphaned-recovered or historical, not deleted-and-forgotten — it's parked in `recovered-project-scoping/` specifically so it can be re-evaluated.

## 3. Callers That Don't Use the Registry-Only Design

Per the current code comment in `project-context.ts` ("Do not reintroduce a second resolver"), everything is supposed to go through `ProjectContext`. Two exceptions exist today, both flagged by the audit as P1 consistency violations:

- **`refactor_batch.ts`** — six call sites (`packages/zenith-mcp/src/tools/refactor_batch.ts:290, 357, 582, 896, 1152, 1179`) do `throw new Error("No project root.")` instead of falling back to the global DB. This violates the global-fallback design (#13): a tool that can't resolve a project root should degrade to global scope, not hard-fail the whole call.
- **`edit_file.ts:96`** and **`stash_restore.ts:157`** — both use `pc.getRoot(path) || path.dirname(path)`. This silently falls back to the *file's parent directory* as a pseudo-project-root when `ProjectContext` returns null, rather than falling back to the global DB like the rest of the system. This is exactly the "fragmented DB resolution" issue the audit and fix-plan (Issue 10) called out for the old direct-`findRepoRoot` pattern — the bypass mechanism changed (now goes through `pc.getRoot` first) but the broken fallback-to-`dirname` behavior was carried over instead of being fixed.

Both patterns should be reconciled with the single global-fallback contract before any new detection mechanism is reintroduced underneath `ProjectContext`.
